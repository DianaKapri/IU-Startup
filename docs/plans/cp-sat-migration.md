# План работ: миграция генератора на CP-SAT (Google OR-tools)

Документ описывает декомпозицию работ по замене текущего самописного CSP-решателя
на промышленный **CP-SAT solver из Google OR-tools** для гарантированно
оптимального построения школьного расписания.

---

## Зачем

Текущий генератор (CSP backtracking + intra-day swap) даёт стабильно работающее
расписание за < 100 мс, но:

- **Не оптимизирует глобально** — застревает в локальном минимуме
- **Не доказывает оптимальность** — не знает, можно ли сделать лучше
- **Cross-day balancing проблематично** — соседние дни нельзя свободно переставлять
- **Soft constraints учитываются ad hoc** — через жадный swap, не через объективную функцию

CP-SAT решает все эти проблемы:

- **Полный перебор пространства решений** с интеллектуальным prunning
- **Гарантия оптимальности** при достаточном времени (минуты вместо мс)
- **Soft constraints как часть objective function** с весами
- **Доказательство infeasibility** — если 0 нарушений невозможно, скажет

## Архитектура

```
┌─────────────────────────────────────────────────┐
│                  Frontend                       │
│   Excel-шаблон → POST /api/generate/from-xlsx   │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│              Backend Node.js                    │
│   • Парсер шаблона (как сейчас)                 │
│   • Валидация → формирование model_input        │
│   • Subprocess или HTTP вызов к Python-сервису  │
└────────────────────┬────────────────────────────┘
                     │ JSON через stdin/stdout (или POST)
                     ▼
┌─────────────────────────────────────────────────┐
│           Python-сервис (cp-sat-solver/)        │
│   • Получает model_input (classes, curriculum,  │
│     teachers, rooms, constraints)               │
│   • Строит CP-SAT model:                        │
│     - Variables: lesson(class, day, slot, ...)  │
│     - Hard constraints: AddBoolOr / AddImplication│
│     - Soft constraints: minimise weighted sum   │
│   • cp_model.CpSolver().Solve()                 │
│   • Возвращает schedule + audit + статус        │
└─────────────────────────────────────────────────┘
```

**Преимущества схемы:**
- Python изолирован — не ломает Node
- Можно деплоить отдельно (отдельный Docker container)
- Легко обновлять/масштабировать

## Зависимости и риски

### Технические

1. **Размер Docker-образа:** ortools + Python ~250-400 МБ. Если ставим в основной образ, он растёт с текущих ~200 МБ до ~600 МБ. Альтернатива — отдельный сервис.
2. **Память:** CP-SAT требует много RAM на больших задачах. Для 44 классов на базовой школе нужно ~1 ГБ. На текущем VPS (1-2 ГБ) — впритык. Может потребоваться апгрейд тарифа.
3. **Время на больших школах:** для 50+ классов с тяжёлыми constraints решение может не сходиться за 30 мин. Тогда возвращать «лучшее найденное» с предупреждением.

### Продуктовые

1. **Пользователь ждёт 30 мин:** UX challenge — нужно показывать прогресс, иметь возможность отменить, делать в фоне.
2. **Сравнение с «быстрым»:** обязательно показывать оба варианта (наш + CP-SAT) и доказывать что CP-SAT лучше — иначе пользователь выберет «быстрый».
3. **Подписка:** 30-минутное вычисление дорого по CPU. Может стать reason for paid tier.

### Команда

1. **Python-стек добавляется:** нужен разработчик с опытом Python и хотя бы базово CP/OR-tools.
2. **DevOps:** деплой стал двух-стековым (Node + Python). Усложняет отладку.

## Альтернативы и их отказ

| Альтернатива | Почему не выбрана |
|--------------|-------------------|
| **MiniZinc / Gecode** | Меньше документации на русском, экосистема меньше |
| **CPLEX / Gurobi** | Платные, лицензия для школ дорогая |
| **Чистый ILP (CBC, GLPK)** | Хуже для логических ограничений вроде «no gaps»; CP-SAT удобнее |
| **Самописный SA / Tabu** | Не доказывает оптимальность; тонкая настройка весов |
| **WebAssembly или JS-биндинги к ortools** | Незрелые, ограниченные возможности; легче subprocess Python |

## Метрики успеха

После полной реализации:

1. На тестовом шаблоне (44 класса) **score ≥ 75/100** (с текущих 0/100) при 5-минутном бюджете.
2. На простых школах (1-10 классов) **score = 100/100** за < 10 секунд.
3. Solver выдаёт **доказательство оптимальности** или предел («решение лучше N не существует при данных constraints»).
4. Пользователь имеет **выбор**: «Быстро (1 сек, score F-D)» или «Качественно (5-30 мин, score A-B)».
