# План работ: генератор расписания

Документ описывает **декомпозицию** работ по развитию генератора расписания — от текущего MVP до продукта, пригодного к продаже школам.

---

## Текущее состояние (Этап 0 — сделано)

**Работает:**
- CSP-решатель с backtracking (`projects/services/generator/index.js`)
- 5 жёстких правил СанПиН (C-01, C-02, C-03, X-01, CONF) — соблюдаются всегда
- 4 мягких правила (D-01, E-01, E-02, E-03) — учитываются в swap-оптимизаторе
- REST API `POST /api/generate`, `GET /api/generate/health`
- Интеграция с фронтом: кнопка «Сгенерировать» в кабинете и визард на главной
- Скоринг + перевод в оценку (`services/audit/scoring.js`)

**Что не работает / ограничения:**
- Score низкий (F при 0 жёстких нарушений): мягкая оптимизация слабая
- Нет учёта специализированных кабинетов
- **Нет проверки вместимости кабинета под количество учеников в классе**
- Нет ограничений по доступности учителей
- Одна смена, 5/6 дней, без вариаций
- Результат не сохраняется в БД
- Нет экспорта в Excel/PDF
- **Ввод данных идёт через тесные textarea на фронте.** Есть готовый Excel-шаблон (`projects/frontend/assets/generator-template.xlsx`, 6 листов: Учебный план, Учителя, Распределение, Ограничения, Кабинеты, Справочники), но фронт/бэк пока не умеют его читать

## Риски и открытые вопросы

1. **Качество swap-оптимизатора.** Нет гарантии, что даже Simulated Annealing поднимет score до A на всех школах — возможно, потребуется переход к более сложным алгоритмам (MILP, ILP-solver через OR-tools).
2. **Производительность при >20 классах.** Текущий лимит 5M BT-вызовов покрывает с запасом, но при добавлении hard-правил (E-04, 1.3.2) может просесть. Нужны нагрузочные тесты на этапе 1.3.
3. **Формат «правильного» Excel-шаблона.** У разных школ разные требования (Минобр, региональные). Нужно определиться, делать один универсальный или набор под регионы.
4. **Интеграции с журналами (4.2.1).** API Дневник.ру/ЭЖД могут быть закрыты для третьих сторон — нужно уточнить на раннем этапе.
5. **Legal/безопасность данных.** С момента, когда храним полные учебные планы с ФИО учителей, — это персональные данные. Нужно добавить согласие на обработку в onboarding (отдельная задача, не включена в WBS).
